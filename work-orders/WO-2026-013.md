# WO-2026-013: Add `subagent_kill` hard-kill tool

## Task Summary

**One-sentence description**: Add a new `subagent_kill` tool that immediately terminates a stuck subagent's underlying process via SIGTERM→SIGKILL, bypassing the soft-stop dance that `subagent_stop` performs when the subagent's RPC pipe is no longer being read.

**Goal**: Give the orchestrator a way to recover from a genuinely hung subagent — one where the child `pi` process is alive but its RPC stdin is not being consumed (deadlock, infinite loop, blocked provider call, etc.) — without waiting up to 5 minutes for the existing `subagent_stop` timeout to fire and without depending on the child being responsive to a steer message.

## Background / Motivation

Today `subagent_stop` (`extensions/subagent-async/index.ts:1941`) works in two phases:

1. Send a steer `"Wrap up..."` message via RPC stdin.
2. Wait up to `STOP_TIMEOUT_MS = 5 * 60 * 1000` for the child to finish on its own (it exits when the model emits a text-only `message_end`).
3. If the timeout fires, fall back to `SIGTERM` then `SIGKILL` after `HARD_KILL_DELAY_MS = 5000`.

This is the right design when the subagent is making progress but should stop — the steer lets it write a useful summary before exit.

It is the **wrong** tool when the subagent is **genuinely stuck**: the steer never gets picked up because the child is no longer reading stdin (or the agent loop has wedged). The orchestrator is then stuck waiting 5 minutes for the hard-kill fallback to fire, while the orchestrator's own turn is blocked on `await subagent_stop(...)`. There is no way to short-circuit.

`subagent_kill` is the missing escape hatch: send `SIGTERM` immediately, follow with `SIGKILL` after a short grace period if the child is still alive, and return control to the orchestrator's turn without waiting for the close handler to fire.

## Implementation Specification

### Tool registration

Add a `SubagentKillParams` TypeBox schema and a `subagent_kill` tool registration immediately after the existing `subagent_stop` registration in `extensions/subagent-async/index.ts`. Schema:

```ts
const KillParams = Type.Object({
    session_id: Type.String({ description: "Session ID of the running subagent to hard-kill" }),
});
```

Tool description (this exact text — orchestrators will read it):

> Hard-kill a stuck subagent by terminating its underlying `pi` process. Use this when `subagent_stop` has not worked because the subagent is genuinely stuck and no longer reading its RPC stdin (e.g. deadlocked, infinite loop, blocked provider call). Sends SIGTERM, then SIGKILL after a short grace period if the child is still alive. The result is delivered as a user message with a clear marker so the orchestrator can see the kill happened. This is destructive — any in-flight work in the subagent is lost. Prefer `subagent_stop` when the subagent is responsive.

### Tool body

The execute function:

1. Look up `running.get(params.session_id)`. If not found, return text-only content: `No running subagent found with session "${params.session_id}". It may have already finished or been stopped.` (Same wording style as `subagent_status` for consistency.)
2. Capture the `rs` reference in a local. Mark `rs.killedExplicitly = true` (new field, see below). Do **not** call `rs.resolveOnStop` — the close handler owns delivery.
3. Cancel the stalled-turn watchdog if present (`rs.staleTimer`).
4. If `rs.proc.exitCode === null && !rs.proc.killed`, send `rs.proc.kill("SIGTERM")`.
5. After `HARD_KILL_DELAY_MS` (reuse the existing constant), if the process is still alive (`!rs.proc.killed && rs.proc.exitCode === null`), send `rs.proc.kill("SIGKILL")`. Use a `setTimeout` so the tool returns immediately rather than blocking.
6. Return text-only content: `${rs.agentName} (${params.session_id}) sent SIGTERM (SIGKILL will follow in ${HARD_KILL_DELAY_MS / 1000}s if still alive). Result will be delivered via the close handler.`

The tool **does not** wait for `proc.on("close", ...)` to fire. It returns immediately. The orchestrator's turn is unblocked as soon as the kill signals are dispatched.

### `RunningSubagent` interface change

Add one field:

```ts
killedExplicitly: boolean;
```

Initialize to `false` in the spawn function (where other defaults like `resolveOnStop: null, isDone: false` are set — `index.ts:586-588`).

### Close handler change

In `proc.on("close", async (code) => {...})` at `index.ts:963-1010`:

1. When computing the footer label (`const how = rs.isDone ? "Completed" : (code === 0 ? "Exited" : "Stopped")`), insert a new branch: if `rs.killedExplicitly`, the footer should read `Killed` (a new style class is fine — pick a new color from the existing `S` style palette, e.g. `S.doneFail` is acceptable for v1, or add `S.killed` if the existing palette is sufficient).
2. In `deliverResult` / `resolveOnStop` branch, prepend a marker line to the delivered message when `rs.killedExplicitly` is true:

   ```
   [Killed via subagent_kill — process terminated, work in this subagent is lost]
   <existing final output>
   ```

3. Do not change `cleanupWorktree` behavior — worktree cleanup should still run. The user may want the auto-commit even on hard kill so they can inspect what was done before the kill.

### Logging

Append a single line to the live log (`logEntry`) immediately after `rs.killedExplicitly = true`:

```
── Killed via subagent_kill (SIGTERM sent, SIGKILL in 5s if needed) ──
```

This makes the kill visible in `/watch <id>` and the persisted log file even if the close handler is slow to fire.

### Tests

Optional. If the implementer wants to add a smoke check, extend `test-resume.cjs` with an integration-gated test under `PI_ASYNC_INTEGRATION=1` that:

- Spawns a long-running subagent (use `mock-slow-agent.md` or similar; if none exists, write a one-shot task that calls `bash sleep 600`).
- Waits for the subagent to register in `running`.
- Calls `subagent_kill` via the tool's execute function with a fake signal.
- Asserts the process exits within ~6 seconds with a non-zero exit code.
- Asserts the meta file's `killedExplicitly` flag would be set (only testable if the helper is extracted from the inline tool body; otherwise skip).

If no helper extraction makes sense, skip this entirely. The orchestrator will smoke-test by dispatching a subagent and killing it from a follow-up turn.

## Invariants

### Cross-file conventions

- New tool follows the existing `pi.registerTool({ name, label, description, parameters, async execute(...) })` pattern.
- Schema uses `Type.Object({...})` from `typebox`, matching the other tools in this file.
- Tool description style (sentence-case, "Use this when X. Does Y. Prefer Z when W.") matches `subagent_stop` and `subagent_status`.
- New `killedExplicitly` field on `RunningSubagent` is initialized at the same code site as `resolveOnStop` and `isDone` (`index.ts:586-588`).
- Reuse existing constants (`HARD_KILL_DELAY_MS`) rather than introducing new ones.

### Default values to preserve

- `HARD_KILL_DELAY_MS` remains `5000` ms.
- `STOP_TIMEOUT_MS` and the existing `subagent_stop` fallback behavior are unchanged. Both tools remain independent.
- The stalled-turn watchdog (`SUBAGENT_STALE_TURN_MS = 5 * 60 * 1000`) is preserved as a "wake the caller" signal, not a kill signal. `subagent_kill` cancels the watchdog timer on the killed session but does not change the watchdog's semantics for other sessions.

### Ordering assumptions

- `rs.killedExplicitly = true` must be set **before** `proc.kill(...)` is called, so the close handler sees the flag when it fires.
- `proc.kill("SIGKILL")` (if needed) must be scheduled via `setTimeout` so the tool's `execute` returns immediately and does not block the orchestrator's turn.

### Error handling conventions

- If `session_id` is not found in the running map, return a text-only result (not an error). Same convention as `subagent_status` for unknown ids.
- If `proc.kill()` throws (extremely unlikely on a live `ChildProcess` but possible), swallow the throw and log to the live log. The tool should not crash on an already-dead proc.
- If `rs.staleTimer` is undefined or already cleared, the `clearTimeout` call must not throw. Guard with `if (rs.staleTimer)`.

### Unspecified invariants (implicit — implementer must reason from the codebase)

- The exact style class to use for the `Killed` footer in the close handler is not specified. Pick the closest existing one or add a new one if it improves clarity. Do not over-engineer.
- The exact wording of the marker line prepended to the delivered result is not specified beyond the prefix `[Killed via subagent_kill`. Keep it terse.
- Whether `subagent_kill` should be added to the `allowedSubagents` allowlist of `implement-flash`/`implement-pro` agents is a follow-up decision — out of scope for v1.
- Whether to add tests is left to the implementer — see "Tests" section.

### Input shapes that must be accepted

- Partial session id (last 8+ chars) — the implementer should attempt the same suffix-match as `subagent_status`. If matching a partial id requires a new helper (not currently shared), the implementer may either (a) reuse an existing helper if one exists, (b) call out the gap in the completion report and add a minimal `resolveSubagentMeta`-style lookup inline, or (c) require a full id in v1 and note the limitation. **Preference: option (a) if a helper exists; otherwise (c) — keep v1 minimal.**

### Behavior matrix

| Case | Expected |
|---|---|
| Session id not in running map | Text-only result: "No running subagent found with session ...", no exception, no proc kill. |
| Session running, proc alive | `SIGTERM` dispatched immediately, `SIGKILL` scheduled in 5s. Tool returns within milliseconds. Close handler eventually fires with `killedExplicitly = true`. Delivered result includes `[Killed via subagent_kill ...]` marker. Footer shows `Killed`. Worktree cleanup still runs. |
| Session running, proc already dead | No-op on signal send (skip the `proc.kill` call). Close handler may have already fired or will fire imminently. Tool returns text saying kill signals were skipped. |
| Session running, SIGTERM is ignored (proc still alive after 5s) | `SIGKILL` dispatched after 5s. Close handler fires shortly after. |
| `proc.kill` throws | Tool does not crash; logs the error to the live log; returns a text-only result indicating the failure. |
| Two `subagent_kill` calls in quick succession on the same session | Idempotent: second call sees `rs.killedExplicitly === true` and either no-ops or returns "Already killed" text. (Implementer's choice on the second-call message — keep it short.) |
